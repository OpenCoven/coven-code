## Part I — Architecture Decision Record

### ADR-1: One surface, two processes (NOT one binary)

**Decision:** `coven` is the singular CLI. coven-code becomes the **Coven engine**: an internal, separately-compiled binary that coven manages end-to-end. All integration crosses a process boundary (exec/spawn/stream-json/HTTP-over-UDS). No Cargo dependency from `coven-cli` on any `claurst-*` crate, ever.

**Why not a single linked binary:**

1. **License.** coven is MIT (`coven/Cargo.toml:7`); coven-code is GPL-3.0 (`coven-code/src-rust/Cargo.toml:22`), inherited from upstream Claurst whose copyright OpenCoven does not wholly own — relicensing is not unilaterally possible. Statically linking `claurst-*` crates into `coven` makes the combined binary a GPL derivative work, forcing `coven` itself to GPL-3.0. A process boundary (exec at arm's length) keeps each work under its own license. This is the ripgrep-in-VS-Code / containerd-under-Docker pattern.
2. **Upstream merges.** `coven-code/COVEN.md` deliberately preserves `claurst-*` crate names so `git merge upstream/main` stays low-friction. Moving the source into the coven workspace kills that.
3. **Dependency and runtime split.** coven-cli is deliberately synchronous (no tokio) with ratatui 0.30 / rusqlite 0.40; the engine is tokio 1.44 / ratatui 0.29 / rusqlite 0.31 across ~148k LoC. Unifying buys nothing users can see.

**What "singular" means instead:** users install one package, type one command, read one doc site, see one brand, hit one config root, and get one session ledger. The engine binary becomes an implementation detail like a bundled shared library — versioned, verified, and invisible.

**Revisit trigger:** see Phase 6 gate. The process-boundary design keeps a future source merge possible without blocking anything now.

### ADR-2: Naming

- User-facing name: **Coven** / command `coven`. The name "coven-code" disappears from install instructions, docs, and error messages (replaced by "the Coven engine").
- Binary name stays `coven-code` internally (upstream-merge-friendly, and PATH installs keep working), installed under `~/.coven/engine/<version>/`.
- Env var canon: `COVEN_CODE_*` remains the engine's namespace; coven passes context via `COVEN_PARENT=coven`, `COVEN_HOME`, `COVEN_DAEMON_SOCKET`.

### ADR-3: Versioning & pinning

Each coven release pins an exact engine version + per-platform SHA-256 in `engine.lock` (checked into `[coven]`, generated by script). `coven` verifies the resolved engine's version at delegation time: warn on drift from pin, hard-fail below `MIN_ENGINE_VERSION`. Engine releases trigger an automated bump PR against coven's `engine.lock`.

### Target command tree (end state)

```
coven                        → engine interactive TUI (auto-installs engine on first run)
coven "fix the login bug"    → cast free-text task (defaults to engine harness)
coven chat | tui             → same TUI (aliases, kept)
coven run <harness> [...]    → daemon-ledgered session: codex | claude | coven-code | ...
coven sessions|attach|kill|… → session ledger (includes engine sessions)
coven daemon …               → daemon lifecycle (unchanged)
coven auth …                 → engine passthrough (Anthropic/Codex credentials)
coven models …               → engine passthrough (model listing)
coven acp                    → engine passthrough (ACP stdio server, for Zed etc.)
coven engine status|install|pin → managed-engine admin
coven upgrade                → upgrade coven + engine to pinned versions
coven code [anything…]       → raw passthrough escape hatch to the engine CLI
coven doctor                 → covers daemon + harnesses + engine + auth
```

**Collision policy:** coven-owned names always win (`run`, `sessions`, `daemon`, …). Engine features are surfaced either as curated passthroughs (`auth`, `models`, `acp`) or via the `coven code` escape hatch. New engine subcommands are reachable through `coven code <sub>` on day one without a coven release.

//! Managed engine (coven-code) resolution and version gating.
//!
//! LICENSE BOUNDARY: the engine is GPL-3.0; coven is MIT. The engine is
//! always a separate process launched by path — never a Cargo dependency.
//! Do not add any claurst-* crate to this workspace.

use anyhow::{anyhow, bail, Context, Result};
use std::ffi::OsStr;
use std::path::{Path, PathBuf};
use std::process::Command;

#[cfg(windows)]
pub const ENGINE_BIN_NAME: &str = "coven-code.exe";
#[cfg(not(windows))]
pub const ENGINE_BIN_NAME: &str = "coven-code";

/// Oldest engine this coven build can drive (contract v1 surfaces).
pub const MIN_ENGINE_VERSION: (u64, u64, u64) = (0, 6, 1);

#[derive(Debug)]
pub enum EngineSource {
    EnvOverride, // COVEN_ENGINE_BIN
    Managed,     // ~/.coven/engine/<current>/
    PathLookup,  // coven-code on PATH
    LegacyHome,  // ~/.coven-code/bin/ (pre-unification installs)
}

#[derive(Debug)]
pub struct ResolvedEngine {
    pub path: PathBuf,
    pub source: EngineSource,
}

pub fn resolve() -> Option<ResolvedEngine> {
    let env_override = std::env::var_os("COVEN_ENGINE_BIN");
    let path_var = std::env::var_os("PATH");
    let home = dirs_next::home_dir();
    resolve_from(env_override.as_deref(), path_var.as_deref(), home.as_deref())
}

pub fn resolve_from(
    env_override: Option<&OsStr>,
    path_var: Option<&OsStr>,
    home: Option<&Path>,
) -> Option<ResolvedEngine> {
    if let Some(p) = env_override {
        let path = PathBuf::from(p);
        if is_executable(&path) {
            return Some(ResolvedEngine { path, source: EngineSource::EnvOverride });
        }
    }
    if let Some(home) = home {
        let engine_root = home.join(".coven").join("engine");
        if let Ok(current) = std::fs::read_to_string(engine_root.join("current")) {
            let path = engine_root.join(current.trim()).join(ENGINE_BIN_NAME);
            if is_executable(&path) {
                return Some(ResolvedEngine { path, source: EngineSource::Managed });
            }
        }
    }
    if let Some(path) = find_on_path(path_var, ENGINE_BIN_NAME) {
        return Some(ResolvedEngine { path, source: EngineSource::PathLookup });
    }
    if let Some(home) = home {
        let path = home.join(".coven-code").join("bin").join(ENGINE_BIN_NAME);
        if is_executable(&path) {
            return Some(ResolvedEngine { path, source: EngineSource::LegacyHome });
        }
    }
    None
}

/// Resolve or produce the single actionable "engine missing" error.
pub fn require() -> Result<ResolvedEngine> {
    resolve().ok_or_else(|| {
        anyhow!(
            "The Coven engine is not installed.\n\n  Run: coven engine install\n\n\
             (or set COVEN_ENGINE_BIN to an existing coven-code binary)"
        )
    })
}

pub fn engine_version(binary: &Path) -> Result<(u64, u64, u64)> {
    let out = Command::new(binary)
        .arg("--version")
        .output()
        .with_context(|| format!("failed to run {} --version", binary.display()))?;
    if !out.status.success() {
        bail!("{} --version exited nonzero", binary.display());
    }
    let text = String::from_utf8_lossy(&out.stdout);
    parse_version_output(&text)
        .ok_or_else(|| anyhow!("unparseable engine version output: {text:?}"))
}

pub fn parse_version_output(text: &str) -> Option<(u64, u64, u64)> {
    let raw = text.split_whitespace().last()?;
    let mut parts = raw.trim_start_matches('v').splitn(3, '.');
    let major = parts.next()?.parse().ok()?;
    let minor = parts.next()?.parse().ok()?;
    // Tolerate suffixes like "1-rc1" on the patch component.
    let patch_raw = parts.next()?;
    let patch = patch_raw
        .split(|c: char| !c.is_ascii_digit())
        .next()?
        .parse()
        .ok()?;
    Some((major, minor, patch))
}

pub fn version_meets_minimum(v: (u64, u64, u64)) -> bool {
    v >= MIN_ENGINE_VERSION
}

fn is_executable(path: &Path) -> bool {
    #[cfg(unix)]
    {
        use std::os::unix::fs::PermissionsExt;
        path.metadata()
            .map(|m| m.is_file() && m.permissions().mode() & 0o111 != 0)
            .unwrap_or(false)
    }
    #[cfg(not(unix))]
    {
        path.is_file()
    }
}

fn find_on_path(path_var: Option<&OsStr>, name: &str) -> Option<PathBuf> {
    let path_var = path_var?;
    std::env::split_paths(path_var)
        .map(|dir| dir.join(name))
        .find(|candidate| is_executable(candidate))
}

```rust
//! Engine download/verify/install. The release ships COMPRESSED ARCHIVES
//! (.tar.gz on unix, .zip on windows), each containing a single `coven-code`
//! binary at the root. Flow: download archive (curl) -> verify archive
//! sha256 (sha2 crate) -> extract inner binary (tar) -> atomically install.
//! Shelling out to curl+tar avoids adding archive/http Rust crates and
//! matches the official install.sh. The pinned checksum is of the ARCHIVE.

use anyhow::{bail, Context, Result};
use sha2::{Digest, Sha256};
use std::path::{Path, PathBuf};
use std::process::Command;

/// Release asset name for a platform. VERIFIED against v0.6.1: archives, not
/// bare binaries — `.tar.gz` on unix, `.zip` on windows.
pub fn artifact_name(os: &str, arch: &str) -> String {
    match os {
        "windows" => format!("coven-code-windows-{arch}.zip"),
        _ => format!("coven-code-{os}-{arch}.tar.gz"),
    }
}

/// Name of the single binary inside every archive (archive root).
pub fn inner_binary_name(os: &str) -> &'static str {
    if os == "windows" { "coven-code.exe" } else { "coven-code" }
}

pub fn release_url(version: &str, artifact: &str) -> String {
    format!("https://github.com/OpenCoven/coven-code/releases/download/v{version}/{artifact}")
}

pub fn current_platform() -> Result<(&'static str, &'static str)> {
    let os = match std::env::consts::OS {
        "macos" => "macos",
        "linux" => "linux",
        "windows" => "windows",
        other => bail!("unsupported platform for managed engine install: {other}"),
    };
    let arch = match std::env::consts::ARCH {
        "x86_64" => "x86_64",
        "aarch64" => "aarch64",
        other => bail!("unsupported architecture: {other}"),
    };
    Ok((os, arch))
}

pub fn install(version: &str, expected_sha256: Option<&str>, force: bool) -> Result<PathBuf> {
    let home = dirs_next::home_dir().context("cannot determine home directory")?;
    let engine_root = home.join(".coven").join("engine");
    let dest_dir = engine_root.join(version);
    let dest = dest_dir.join(crate::engine::ENGINE_BIN_NAME);
    if dest.exists() && !force {
        activate(&engine_root, version)?;
        return Ok(dest);
    }
    let (os, arch) = current_platform()?;
    let artifact = artifact_name(os, arch);
    let url = release_url(version, &artifact);

    std::fs::create_dir_all(&dest_dir)
        .with_context(|| format!("creating {}", dest_dir.display()))?;

    // 1. Download the archive to a temp path in the destination dir.
    let archive = dest_dir.join(&artifact);
    let status = Command::new("curl")
        .args(["-fSL", "--retry", "3", "-o"])
        .arg(&archive)
        .arg(&url)
        .status()
        .context("failed to run curl (required for engine install)")?;
    if !status.success() {
        let _ = std::fs::remove_file(&archive);
        bail!("download failed: {url}");
    }

    // 2. Verify the ARCHIVE checksum BEFORE extraction (fail closed on mismatch).
    let digest = hex_digest(&std::fs::read(&archive)?);
    match expected_sha256 {
        Some(expected) if !digest.eq_ignore_ascii_case(expected) => {
            let _ = std::fs::remove_file(&archive);
            bail!(
                "checksum mismatch for {artifact}: expected {expected}, got {digest}. \
                 Refusing to install."
            );
        }
        None => eprintln!(
            "warning: no pinned checksum for {artifact}; installing unverified (dev mode)"
        ),
        _ => {}
    }

    // 3. Extract the single inner binary into a staging path, then atomically
    //    rename into place. bsdtar (system `tar`) handles both .tar.gz and .zip.
    let stage_dir = dest_dir.join(".stage");
    let _ = std::fs::remove_dir_all(&stage_dir);
    std::fs::create_dir_all(&stage_dir)?;
    let untar = Command::new("tar")
        .arg("-xf")
        .arg(&archive)
        .arg("-C")
        .arg(&stage_dir)
        .status()
        .context("failed to run tar (required to unpack the engine archive)")?;
    if !untar.success() {
        let _ = std::fs::remove_file(&archive);
        let _ = std::fs::remove_dir_all(&stage_dir);
        bail!("failed to extract {artifact}");
    }
    let extracted = stage_dir.join(inner_binary_name(os));
    if !extracted.is_file() {
        let _ = std::fs::remove_file(&archive);
        let _ = std::fs::remove_dir_all(&stage_dir);
        bail!(
            "archive {artifact} did not contain the expected binary {}",
            inner_binary_name(os)
        );
    }

    #[cfg(unix)]
    {
        use std::os::unix::fs::PermissionsExt;
        std::fs::set_permissions(&extracted, std::fs::Permissions::from_mode(0o755))?;
    }
    std::fs::rename(&extracted, &dest)?; // atomic within the same dir

    // 4. Cleanup: drop the archive and staging dir; publish the `current` pointer.
    let _ = std::fs::remove_file(&archive);
    let _ = std::fs::remove_dir_all(&stage_dir);
    activate(&engine_root, version)?;
    Ok(dest)
}

fn activate(engine_root: &Path, version: &str) -> Result<()> {
    // `current` is a plain text file (not a symlink) so the scheme is
    // identical on Windows; written via temp+rename for atomicity.
    let tmp = engine_root.join("current.tmp");
    std::fs::write(&tmp, version)?;
    std::fs::rename(tmp, engine_root.join("current"))?;
    Ok(())
}

fn hex_digest(bytes: &[u8]) -> String {
    let mut hasher = Sha256::new();
    hasher.update(bytes);
    hasher
        .finalize()
        .iter()
        .map(|b| format!("{b:02x}"))
        .collect()
}
```

#!/usr/bin/env bash
# Usage: scripts/pin-engine.sh <engine-version>
# Downloads each release asset for OpenCoven/coven-code v<version>,
# computes sha256, and rewrites crates/coven-cli/engine.lock.
set -euo pipefail
VERSION="${1:?usage: pin-engine.sh <version>}"
LOCK="crates/coven-cli/engine.lock"
ASSETS=$(gh release view "v${VERSION}" --repo OpenCoven/coven-code --json assets -q '.assets[].name')
{
  echo '# Generated by scripts/pin-engine.sh — do not edit by hand.'
  echo "version = \"${VERSION}\""
  echo 'contract = 1'
  echo
  echo '[sha256]'
  for asset in $ASSETS; do
    case "$asset" in
      coven-code-*)
        tmp=$(mktemp)
        gh release download "v${VERSION}" --repo OpenCoven/coven-code --pattern "$asset" --output "$tmp"
        sum=$(shasum -a 256 "$tmp" | awk '{print $1}')
        rm -f "$tmp"
        echo "\"${asset}\" = \"${sum}\""
        ;;
    esac
  done
} > "$LOCK"
echo "pinned engine ${VERSION} in ${LOCK}"

Key tasks:

1. **Engine home relocation** `[coven-code]` — default engine home becomes `~/.coven/code/` when `COVEN_HOME`/parent context is present; `COVEN_CODE_HOME` env override preserved. Migration on first run under the new default: if `~/.coven-code/` exists and `~/.coven/code/` doesn't → move contents, leave a symlink at `~/.coven-code` → `~/.coven/code` for one deprecation cycle (Windows: junction). All path construction already flows through `dirs`-based helpers in `claurst-core` — audit `config.rs`, `auth_store.rs`, `session_storage.rs`, `oauth.rs` for hardcoded `.coven-code` strings first.
2. **Config layering** `[coven-code]` — settings load order becomes: `~/.coven/settings.json` (shared keys: default model, theme, permission defaults) → `~/.coven/code/settings.json` → project settings. Shared-key schema documented in the contract doc.
3. **Auth surface** `[coven]` — `coven doctor` merges engine auth (`auth status --json`) and harness availability into one "Credentials" panel. **Hard rule carried forward:** the engine never imports/replays OAuth tokens from Claude Code or other CLIs (`coven-code/AGENTS.md` — the import path was deliberately removed; it triggers rate-limiting). Unification means *presentation*, never token sharing.
4. **Cross-session search** `[coven]` — daemon indexes notifier-registered transcript paths: a background task reads new JSONL entries into the existing FTS5 events table (kind: `transcript_text`), bounded by the existing retention policy. `coven sessions search` then spans daemon-run and TUI sessions.

**Phase 4 exit check:** fresh install writes nothing outside `~/.coven`; existing user upgraded in place with data intact (migration test with a populated fixture home dir); one doctor panel answers "am I logged in, to what, via what".

## Part VIII — Phase 5: Seamless UX & brand (own plan doc at kickoff)

1. **Naming sweep** — "coven-code" removed from all user-facing strings in both repos (READMEs, error messages, docs.opencoven.ai, npm descriptions). The engine repo README repositions as "the Coven engine — contributor guide".
2. **Direct-invocation notice** `[coven-code]` — when invoked with a TTY and without `COVEN_PARENT`, print one dim line: `coven-code is the Coven engine — the supported CLI is 'coven' (npm i -g @opencoven/cli)`. Silent when driven by coven. `coven-cave` alias: same notice + sunset date, removed one minor version later.
3. **Version surface** `[coven]` — `coven --version` prints `coven <v> (engine coven-code <v>, pinned <v>)`.
4. **Theme** `[coven-code]` — default TUI theme adopts the OpenCoven palette per the existing seam (`COVEN.md`: `theme_colors.rs`, violet #8B5CF6 / pink #EC4899); mascot stays.
5. **Completions** `[coven]` — `coven completions` covers the passthrough subcommands; `coven code` completes nothing (opaque passthrough) by design.
6. **npm deprecation** — execute the `npm deprecate` from Task 2.3; engine binaries continue shipping via GitHub Releases (they're the install artifact).

**Phase 5 exit check:** a new user can go install → auth → first agent session → session search without ever reading the string "coven-code".

## Part IX — Phase 6: Decision gate — full source merge

Standing recommendation: **do not merge the source trees.** Revisit only if ALL of:

1. Upstream Claurst merges have stopped being valuable (check: how many upstream merges landed in the last two quarters).
2. OpenCoven is willing to ship the unified binary as GPL-3.0 (MIT `coven` cannot absorb GPL code any other way — relicensing coven-code requires upstream Claurst copyright holders' consent).
3. Someone is funded to do the mechanical unification: ratatui 0.30↔0.29, rusqlite 0.40↔0.31, introducing tokio into the deliberately-sync coven-cli, CI merge, release-pipeline merge (~148k LoC absorbed into a ~35k LoC workspace).

If ever taken: `git subtree add` (history-preserving) into `coven/crates/`, single-binary dispatch replacing the delegation seam, workspace license → GPL-3.0, npm/dist consolidation is already done (Phase 2), and the daemon/notifier/contract work (Phase 3–4) carries over unchanged — which is exactly why the process-boundary design loses nothing while it waits.

## Part X — Cross-cutting

### Testing strategy
- **Unit:** resolver, version parsing, lock parsing, auto-install gating, passthrough collision guard (Phase 1 tasks above).
- **Contract:** `coven/tests/engine_contract.rs` run in both repos' CI against a real binary (Task 2.2) — this is the load-bearing suite; every contract-doc bullet needs a test.
- **Golden transcripts:** recorded stream-json fixtures, re-validated by the reverse CI job on every engine PR.
- **End-to-end:** clean-container install test (Task 2.3); populated-home migration test (Phase 4).
- Both repos keep their existing gates (`cargo fmt --check`, `clippy -D warnings`, `cargo test --workspace`).

### Risks & mitigations

| Risk | Mitigation |
|---|---|
| GPL/MIT boundary erosion (someone links a claurst crate "just for a type") | Comment ban in `engine.rs` header; `deny.toml` addition in `[coven]` denying `claurst-*` + GPL-3.0 licenses in the dependency tree; PROVENANCE.md notes in both repos |
| Version skew (user's PATH engine ≠ pin) | Handshake warning (1.2), pin in doctor (1.6), managed install wins resolution order (1.1) |
| Supply chain on engine download | SHA-256 pinning in `engine.lock` (2.1); fail-closed on mismatch; artifact names verified against real releases (1.3 Step 3) |
| Contract drift breaking users mid-upgrade | Reverse CI job (2.2) blocks engine PRs that break coven@main; `MIN_ENGINE_VERSION` blocks coven from driving too-old engines |
| npm PATH shadowing (seen before on this machine: stale shim shadowed the real binary) | Doctor prints WHICH engine source won and why; `coven engine which` for scripts |
| Windows divergence (no UDS notifier, exec vs spawn, `.exe` names) | Text-file `current` pointer (no symlinks); spawn+wait passthrough; notifier named-pipe variant or documented skip in Phase 3 plan |
| Daemon absent when TUI runs | Notifier is fire-and-forget with all errors swallowed (3.3) — ledger integration is additive, never required |
| Migration data loss (Phase 4) | Move-then-symlink (no deletes); populated-fixture migration test; one-cycle rollback = remove symlink, restore dir name |

### Rollback
Each phase is independently revertible: the legacy resolution order (PATH + `~/.coven-code/bin`) is kept in the resolver permanently, the old npm package keeps working until the Phase 5 deprecation, and the notifier/harness additions are additive. Reverting any phase's commits restores the previous working state without touching user data.

### Sequencing & ownership notes
- Phases 1–2 are pure `[coven]`-side (plus CI in `[coven-code]`) and can start immediately; nothing blocks on engine changes.
- The engine's in-flight work (openai-codex alias, image forwarding, typeahead — currently uncommitted in the working tree) is orthogonal; land it before starting Phase 3's engine-side tasks to keep diffs clean.
- Every commit follows both repos' existing rules: signed (`-S`, verify `git log -1 --show-signature`), scoped `git add <file>` only, version bumps only via `scripts/bump-version.py` in `[coven-code]`.
